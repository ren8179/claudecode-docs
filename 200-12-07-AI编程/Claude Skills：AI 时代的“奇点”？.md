# [Claude Skills：AI 时代的“奇点”？](https://mp.weixin.qq.com/s/tvGbvchOjpgALu_VOv5zlA)

今天这篇，想跟大家聊聊 Claude Skills，主要的原因是，我在深入使用了 Claude Skills 之后，隐约感觉到这似乎是一个 AI 能够真正帮助大众解决实实在在问题的“奇点”时刻。在它之前，AI 只是少数发烧友的专属；在它之后，AI 可以为每个人所用，切切实实地解决实际的问题，用好了提升效率岂止 10x。

## Prompt Engineering 其实门槛很高

这一波 AI 革命的核心特点，是大模型，尤其是大语言模型（LLM）。LLM 的本质，是一个通过预测下一个词元（Next Token Prediction），来把一个序列转换为另一个序列的概率模型。

要用好 LLM，一个核心的技能就是**提示词工程（Prompt Engineering）**，我在之前这篇文章里有详细介绍过[《人人都需要掌握的 Prompt Engineering 技巧》](https://mp.weixin.qq.com/s?__biz=Mzg2NTEwMDU5NA==&mid=2247484162&idx=1&sn=ed691e630680d85e44f7f0f798be0da3&scene=21#wechat_redirect)。

关于提示词工程，Andrej Karpthy 有一个很形象的比喻，我之前在这篇文章里分享过[《Andrej Karpathy: 2025大模型总结》的启示](https://mp.weixin.qq.com/s?__biz=Mzg2NTEwMDU5NA==&mid=2247485001&idx=1&sn=d5a4fea8ad0a616ac195843a9eef1857&scene=21#wechat_redirect)。他说，我们现在通过写提示词来用好 LLM 的过程，很像是在“召唤幽灵”。确实是如此，LLM 中压缩了海量的知识，我们是在通过提示词这个序列，去“召唤”出 LLM 中我们期待的那个序列。

我之前一直有一个洞察：**对于很多人来说，提出一个好问题其实是很难的。所以那些在产品里，要求用户要提出一个好问题，来激发出好答案的形态，往往都是有很高的门槛的**。

这个洞察对于大模型也是成立的，通过写提示词来用好它，也是一件门槛极高的事情。相信写过一些提示词的朋友，都会有这种切身的感受的。

经常在网上看到一些朋友分享一些厉害的提示词，也有人热衷于收藏各种各样的提示词，但问题是，这些都是别人写的，每个人都有自己的问题要解决，你要解决自己的问题，是否也能像别人一样，写出这样优质的提示词呢？

## Context Engineering 其实也很遥远

过去的一年，各种 Agent 类的应用爆发，很多人说 2025 年是 Agent 的元年，Andrej Karpthy 说接下来是 Agent 的 10 年。很多人似乎都在押注 Agent 类应用，好像这个已经形成了一个共识。

我们知道，LLM 是一个语言模型，它只有把一个语言序列转换为另一个语言序列的能力，它并不能解决真实的问题。要想让它解决真实的问题，业界给出的解决方案就是 Agent，Agent 的工作原理大致是这样的：

（1）LLM 负责问题的理解、任务的规划；

（2）根据当前规划，调用工具去执行具体的任务，执行完成把结果反馈给 LLM；

（3）LLM 拿到执行的结果，再结合最新的信息进行下一步的规划；

（4）循环上面的过程，直至最终问题解决。

为了做好 Agent 系统，从去年开始又出现了一个全新的概念，叫**上下文工程（Context Engineering）**。它的本质其实还是提示词工程，而且是更为复杂的提示词工程，它要持续把工具执行的各种状态、结果（也叫 Observation）通过精心的编排形成一个提示词，去召唤大模型中的“幽灵”。

如果说 LLM 只是一个理论水平很高、但没有动手能力的“博士生”，那 Agent 就是这个博士生变成了“科学家”、“工程师”，它真的能够帮助人们解决实实在在的问题了。

但做好 Agent 的关键是上下文工程，而它的门槛又很高，所以对于大众来说，实现自己一些个性化的 Agent 其实是挺难的一件事，核心是因为 Context Engineering 的门槛还是太高了。

## Claude Skills 带来希望

## Claude Commands / Skills / Agents

要聊 Claude Skills，就不得不也一起聊聊它的另外两个兄弟，一个叫 Commands, 一个叫 Agents。这三者，是在 Claude 生态里经常被大家提到的三个。这里特别说明一下，这里说的 Agents，专指 Claude 中的 Agents。

Commands / Skills / Agents 三兄弟，本质上都是提示词工程。只是根据不同的应用场景，分别定义成了三种形态：

（1）Commands 相当于是一个快捷方式，相当于一个“按钮”，定义好了之后，需要的时候 `/cmd` 就可以唤起使用。一些简单的重复性的流程定义成 Commands 就能提效不少

（2）Skills 相当于是一个复杂任务的“说明书”，它定义好了之后，由 LLM 自主判断是否调用它。它的最鲜明的一个特点是叫“渐进式披露”，是说这个说明书本身可以很复杂，但 LLM 只根据任务加载需要的部分，因此也不会一上来就消耗很多的大模型的 Token；

（3）Agents 是更重、更复杂的任务，它的特点是执行的时候可以独立上下文，执行好了直接交付结果。相当于一个在后台默默执行任务的小兄弟，执行好了给结果就好。

根据我的实践，发现日常工作中用 Commands 和 Skills 会比较多，Agents 相对较少。Commands 和 Skills 中，真正提升效率、扩展个人能力边界的，我觉得是 Skills，这也是我特别想写今天这篇文章，来分享关于 Skills 的一些想法的初衷。

## Claude Skills 的创新

Claude Skills，它是一个能够让大众用 AI 解决真实问题的非常优雅的路径，在这个方案中，有几个非常创新的地方：

**第一个创新：通过程序连接数字世界与物理世界。**

我们都知道，目前全球在大模型里面，编程能力最强的一直是 Claude。这是 Claude 背后的公司 Anthropic 在大模型时代的一个 Bet -- 专心做好编程这件事。

文字是我们人类之间交流、沟通的桥梁，而程序则是我们人类跟数字世界沟通的桥梁。拥有最强的能够写程序的大模型，就是一项非常厉害的能力，就能够让 AI 真正去帮人们做事情了，因为现下我们很多的工作都是在数字世界里进行的。

可以说，拥有了能稳定可靠写程序能力的大模型，就拥有了操控数字世界的基石。

**第二个创新：通过 Skills，让 AI 使用不再有门槛。**

我们前面铺垫了很多，其实有一个核心的主题是，要用好 AI，最最基础的动作是写好提示词，但写好提示词对于大众来说还是一件门槛极高的事情，所以如果说要让大众用写好提示词的方式来用好 AI，是一件不太现实的事情。

Claude 定义了 Skills 这种形态，而且很妙的一点是，它们官方提供了一个叫 `skill-creator` 的元 Skills，通过这个元 Skills，我们只需要把我们想做啥大致说清楚，然后它就能引导我们一步步完成一个 Skills 的创建，而且在未来使用的过程中，还可以持续的迭代完善。

相当于`skill-creator`是一个把我们的需求，给翻译成大模型的提示词的利器，有了它写提示词就没那么难了。

把上述两大创新组合在一起，就产生了非常强大的威力。有了给大模型的任务说明书，特定的任务还可以让大模型写程序来解决得更加可靠，这样的组合不就是我们人类社会工作模式的高度抽象么：**一些人在实践中不断摸索最佳实践，一些人通过写程序开发工具来提效，把两者结合到一起，持续迭代进化，就是优质的生产力**。

想象一下，一个拥有无限写程序能力的、还能够不断积累最佳实践、还可以不眠不休的“硅基”伙伴，会给我们带来什么样的体验，会给人类社会带来什么样的影响？

## 如何用好 Skills

如果说要用好 Skills 只给一条建议的话，那必然是：**要多去用，实践出真知**。

具体来说，一些可行的路径是这样的：

第一条路径，可以多去看看别人写的 Skills，把它们装起来，用用看。我特别推荐的是，先去把 Claude 官方提供的 Skills 装起来。

对于很多人来说，日常的工作可能就是跟 Excel、Word、PPT、PDF 等这些文档打交道，Claude 官方就提供了操作这类文档的 Skills，有了它们就可以把很多有点“体力活”性质的工作给自动化了。

比如说，对于财务人员，可能经常要去手动处理一些大型的 Excel 表格。这样的任务，对于人来说耗时、费力不说，还特别容易出错。但对于 AI 来说，有了表格的处理的能力，有了写程序的能力，真的能做到又快、又准。这个是一个我亲测的场景，效果非常明显。

第二条路径，可以自己尝试动手创建一些自己的 Skills，这里因为有 `skill-creator` 的协助，我们只要跟着它的引导一步步往下就好了，比写一个完整好用的提示词要简单很多。

这里的核心是，找到自己的真实需求。拿我自己来说，我们有一个固定动作是写小红书的帖子，这里面涉及到的不只是写内容，写好后还要去制作帖子的图，这个之前是要依赖设计师的，而且每个帖子设计师都要花不少时间。

我把这个任务拆解成了两个 Skills，一个负责制作帖子的正文图，一个负责制作帖子的封面图：

（1）正文图重点是清晰地表达内容：让 AI 通过写程序的方式生成矢量图来表达，这样能更好地把想要表达的信息表达清楚；

（2）封面图要能够吸引用户来看，吸睛程度会比较重要：让 AI 调用当下最好用的生图服务 Nano Banana 的 API，去生成精美的插画，然后再制作成吸睛的封面。

这样一套流程下来，一个帖子写好后，制作图片的过程，就由原来设计师可能要花个把小时，缩短到了十来分钟。最最关键的是，这个流程是稳定可靠的，不再需要“开盲盒”了。

如果你现在还在犹豫如何使用 AI 来提效，那不妨试试 Claude Skills？时代大浪来袭，唯有积极拥抱、纵情一跃，才能不被拍在岸上。
